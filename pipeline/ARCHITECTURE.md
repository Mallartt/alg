# Архитектура пайплайна

## Обзор

Пайплайн принимает на вход MP4-видео с робота, выходит — CSV в формате
`sample.csv`. Между ними четыре последовательных стадии, объединённые
центральным оркестратором.

```
┌────────────┐   ┌────────────┐   ┌────────────────┐   ┌───────────────┐
│   stream   │──▶│   crops    │──▶│   recognize    │──▶│    export     │
│  (sampler  │   │ (geometry, │   │  (filters →    │   │ (CSV writer   │
│   + YOLO   │   │  enhance)  │   │   VLM + QR)    │   │  по схеме     │
│   tracker) │   │            │   │                │   │  sample.csv)  │
└────────────┘   └────────────┘   └────────────────┘   └───────────────┘
       ▲                  ▲                  │                 ▲
       │                  │                  ▼                 │
       │           ┌──────┴────────┐  ┌───────────────┐        │
       │           │  catalog      │  │ core (track   │        │
       │           │  (db_hack.csv)│  │ registry +    │────────┘
       │           └───────────────┘  │ majority vote)│
       │                              └───────────────┘
       │                                      ▲
       │     ┌────────────────────────────────┘
       │     │ Lens undistort (опц.) применяется до YOLO
       └─────┘
```

Снаружи всё доступно двумя способами:

* **CLI** — `run.py` (видео целиком) и `run_image.py` (одиночный кроп);
* **HTTP** — `serve.py` поднимает FastAPI с тремя эндпоинтами:
  `POST /jobs/submit`, `GET /jobs/{id}`, `GET /jobs/{id}/result`.

## Стадии конвейера

### 1. `src/stream/` — захват и детекция

* **`FrameSampler`** — сэмплирует 1–2 кадра/сек из MP4. Тяжёлые 4К кадры,
  которые не идут в работу, пропускаются через `cap.grab()` (без декодирования).
* **`LensUndistorter`** — опционально применяет коэффициенты дисторсии
  камеры (k1, k2, p1, p2, k3) к кадру до детекции. Карты `cv2.remap` строятся
  один раз при инициализации.
* **`PriceTagDetector`** — обёртка над `model.track(...)` Ultralytics:
  YOLOv8 + ByteTrack в одном вызове. Каждое срабатывание выходит наружу как
  `CropEvent` с уже вырезанным изображением, исходной bbox и **centerness**
  (число 0..1, насколько детекция близка к центру кадра).

### 2. `src/crops/` — геометрия и enhance

* `build_crop` режет кадр по xyxy + защитный отступ;
* `_rotate_axis_aligned` поворачивает кроп на 90° влево (или иной угол);
* `deskew_quad` ищет четырёхугольник ценника по Canny + approxPolyDP и
  выпрямляет его через `getPerspectiveTransform` (для скошенных кропов);
* `boost_text_contrast` применяет CLAHE по L-каналу LAB и опционально
  unsharp mask — это и есть «повысить контраст текста, чтобы OCR было легче».
* `upscale_bicubic` делает x2/x3 — даёт VLM больше пикселей для мелкого шрифта.

### 3. `src/recognize/` — обработка одного кропа

`CropRecognizer.__call__(event)` — единая точка входа. Внутри:

1. **`sharpness.is_sharp_enough`** — рубим размытые кадры (variance of
   Laplacian после `NORM_MINMAX`).
2. **`tag_color.classify_tag_color`** — определяем цвет (red/yellow/white/other)
   по гистограмме HSV. Полезно как метаданные и для эвристик.
3. **`deskew_quad`** + **`boost_text_contrast`** + апскейл.
4. **`CodeReader.scan`** — пробуем `pyzbar` → `cv2.barcode` → `cv2.QRCodeDetector`,
   сливаем найденные значения, выкидывая дубликаты.
5. **`LocalVlmClient.ask_about_image`** — отправляем (текстовый промпт + JPEG
   в base64) на локальный llama.cpp-сервер через OpenAI-совместимый API.
   Ответ парсится в JSON (поддерживается markdown-блок и срез по `{...}`).
6. **Валидация** — должно быть либо `product_name`, либо `barcode`, плюс хотя
   бы одна цена. Иначе `RecognitionOutcome.ok = False`.
7. **`ProductCatalog.enrich`** — если LLM не дочитал название, пытаемся
   подтянуть его по штрихкоду из `db_hack.csv` (первые 100 строк).

### 4. `src/core/` — оркестрация и голосование

* **`CropEvent` / `RecognitionOutcome` / `TrackRecord`** — лёгкие dataclass
  для передачи данных между стадиями.
* **`TrackRegistry`** — буфер по `track_id`. Для каждого трека хранит лучший
  по `centerness` кроп и количество выполненных OCR-попыток. Логика:
  * `min_attempts = 1` — минимум одна OCR попытка на трек обязательно;
  * `max_attempts = 3` — больше трёх не делаем;
  * trigger OCR: либо объект уехал из кадра (`stale_after_frames`), либо
    уже хороший центрированный кроп (centerness > 0.85) и есть бюджет
    на дополнительную попытку.
* **`MajorityVoter`** — по каждому полю выбирает наиболее частое значение
  среди успешных попыток. Цены проверяются по приоритету (`price_card` →
  `price_discount` → `price_default`); если хотя бы по одной цене собрался
  кворум — итог помечается как `stable`.
* **`RecognitionFlow`** — главный цикл: на каждое событие зовёт
  `registry.observe`, дренирует «готовые» треки в OCR, в самом конце
  гарантирует, что у каждого трека был хотя бы один проход, и возвращает
  список словарей для CSV.

### 5. `src/export/csv_writer.py`

Сериализует строки в строгом порядке столбцов `sample.csv`. Пустые ячейки
заполняются строкой `нет` (как в эталоне).

### 6. `src/catalog/product_catalog.py`

Читает `db_hack.csv` (cp1251, `;`) — **только первые 100 строк** по
умолчанию (`recognize.catalog_max_rows`). Используется как «справочник»
barcode → product_name. Файл большой, поэтому полностью никогда не
загружаем.

### 7. `src/api/server.py`

FastAPI-сервис с тремя эндпоинтами:

| Метод | URL                       | Что делает                          |
|-------|---------------------------|-------------------------------------|
| POST  | `/jobs/submit`            | Принимает `video` (UploadFile) +    |
|       |                           | `rotation_deg`, `frames_per_second`. |
|       |                           | Кладёт файл в `runtime/uploads/`,   |
|       |                           | запускает фоновую задачу.           |
| GET   | `/jobs/{id}`              | Возвращает `status`: `queued`/      |
|       |                           | `running`/`done`/`failed`.          |
| GET   | `/jobs/{id}/result`       | Скачивает CSV, когда `done`.        |
| GET   | `/health`                 | Технический.                        |

In-memory `dict` хватает для одного инстанса. Если понадобится много
рабочих — оборачивайте в Redis/SQLite, точку расширения видно сразу.

## Жизненный цикл одного физического ценника

1. Робот заезжает в зону → YOLO детектит ценник → ByteTrack присваивает
   `track_id = 17`.
2. Кадр за кадром в `TrackRegistry` обновляется `best_event` (тот, где
   ценник ближе всего к центру).
3. Когда `centerness > 0.85` *или* робот уехал и объект пропал на 30
   кадров — `CropRecognizer` обрабатывает лучший кроп:
   blur → color → deskew → enhance → QR → VLM → catalog.
4. Результат складывается в `outcomes`. Если есть бюджет (1..3 попытки),
   могут быть ещё проходы — например, по другому, ещё более центрированному
   кадру.
5. В конце видео `RecognitionFlow._close_all_tracks` вынуждает минимум
   одну попытку для каждого трека, у которого она ещё не сделана.
6. `MajorityVoter.reduce` склеивает все попытки в **одну строку** —
   именно поэтому в итоговом CSV ровно одна строка на физический ценник.
7. `write_submission` пишет результат, заполняя недостающие поля строкой
   `нет`.

## Расширяемость

* **Другая VLM:** замените `LocalVlmClient` или поменяйте `base_url` в
  `config.toml` (любой OpenAI-совместимый сервер подойдёт).
* **Другие веса YOLO:** переподложите файл и обновите `paths.weights`.
* **Без `pyzbar`:** `CodeReader` сам определит отсутствие и пойдёт
  через OpenCV.
* **Без камерной калибровки:** `lens.enabled = false` (по умолчанию).
* **Изменить кол-во OCR-попыток на трек:** `tracking.min_attempts` /
  `tracking.max_attempts`.
